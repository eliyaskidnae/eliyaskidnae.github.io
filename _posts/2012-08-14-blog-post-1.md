## Introduction

LIO-SAM is a tightly coupled LiDAR-inertial odometry framework designed for accurate real-time state estimation, mapping, and global optimization. It combines LiDAR, IMU, and optionally GPS measurements using a factor graph-based optimization approach.

The main strength of LIO-SAM is that it does not treat LiDAR odometry, IMU integration, loop closure, and GPS correction as isolated components. Instead, these measurements are integrated into a unified optimization framework, making the system suitable for robust localization and mapping in complex environments.

## What is LIO-SAM?

LIO-SAM stands for **LiDAR Inertial Odometry via Smoothing and Mapping**. It is built on top of a factor graph and uses incremental smoothing to optimize the robot trajectory over time.

In simple terms, LIO-SAM uses:

- **LiDAR** to estimate motion and build a map.
- **IMU** to provide high-frequency motion prediction and motion compensation.
- **GPS**, when available, to reduce global drift.
- **Loop closure** to correct accumulated drift when revisiting previously seen places.
- **Factor graph optimization** to fuse all measurements consistently.

This makes LIO-SAM useful for autonomous robots, vehicles, mapping platforms, and multi-sensor localization systems.

## Required Sensors

LIO-SAM usually requires the following sensors:

### LiDAR

Supported LiDAR types include:

- Livox
- Velodyne
- Ouster
- Robosense
- Mulran-compatible point cloud formats

### IMU

The original LIO-SAM framework is designed for a **9-axis IMU**, which includes:

- Accelerometer
- Gyroscope
- Magnetometer or orientation information

Some modified versions support a **6-axis IMU**, using only accelerometer and gyroscope measurements.

### GPS

GPS is optional. When available, it can be added as a factor in the graph to help reduce global drift.

## 1. Image Projection Node

The Image Projection node subscribes to raw IMU data, corrected/preintegrated IMU data, and raw point cloud data. Its main output is a motion-compensated point cloud.

### Main Responsibilities

#### Multi-Sensor Point Cloud Support

Different LiDAR sensors publish point clouds in different formats. The Image Projection node converts these formats into a common point type, usually `PointXYZIRT`, which contains:

- X, Y, Z coordinates
- Intensity
- Ring ID
- Timestamp

This conversion is handled in the `cachePointCloud()` function. The function first checks whether the incoming point cloud contains the required fields, then converts it into the expected common format.

#### Motion Compensation

During one LiDAR scan, the robot may move. This motion causes distortion in the point cloud. The Image Projection node uses IMU information to compensate for this motion and produce a corrected scan.

This corrected scan is important because it improves scan matching accuracy in later stages.

## 2. IMU Preintegration Node

The IMU Preintegration node integrates many high-frequency IMU measurements into a single relative motion estimate. Instead of optimizing every individual IMU measurement separately, the node summarizes them as a preintegrated motion constraint.

This allows LIO-SAM to combine high-frequency IMU data with lower-frequency LiDAR odometry efficiently.

### Inputs

The node subscribes to:

- `imuTopic`: raw IMU data, usually at 200–1000 Hz
- `/mapping/odometry_incremental`: LiDAR odometry from the mapping node, usually around 10 Hz

### Outputs

The node publishes:

- `odomTopic + "_incremental"`: high-frequency odometry estimate based on IMU preintegration

### Why IMU Preintegration is Important

IMU data provides smooth and high-frequency motion estimates, but it drifts over time. LiDAR odometry is usually more accurate over longer periods, but it is published at a lower frequency.

LIO-SAM uses LiDAR odometry to correct IMU drift and estimate IMU bias. This makes the final odometry smoother and more reliable.

### Example Code Concept

```cpp
// Integrate one IMU measurement
imuIntegratorImu_->integrateMeasurement(
    gtsam::Vector3(
        thisImu.linear_acceleration.x,
        thisImu.linear_acceleration.y,
        thisImu.linear_acceleration.z
    ),
    gtsam::Vector3(
        thisImu.angular_velocity.x,
        thisImu.angular_velocity.y,
        thisImu.angular_velocity.z
    ),
    dt
);

// Predict odometry
gtsam::NavState currentState =
    imuIntegratorImu_->predict(prevStateOdom, prevBiasOdom);
```

## 3. Map Optimization Node

The Map Optimization node is the main component responsible for global consistency. While the Image Projection node corrects per-scan distortion and the IMU Preintegration node provides high-rate odometry, the Map Optimization node builds and refines the global map.

### Main Responsibilities

#### Scan-to-Local-Map Matching

Instead of matching only consecutive scans, LIO-SAM matches each new scan against an existing local map. This improves robustness because the local map contains more geometric information than a single previous scan.

The initial guess for scan matching usually comes from IMU preintegration odometry.

#### Keyframe Selection

LIO-SAM does not add every LiDAR frame as a keyframe. Instead, it selects keyframes based on motion thresholds, such as:

- Translation change, for example 0.5 meters
- Rotation change, for example 10 degrees

This reduces computational cost while preserving important map information.

#### Factor Graph Optimization

The system uses a factor graph to combine different constraints, such as:

- LiDAR odometry factors
- IMU preintegration factors
- GPS factors
- Loop closure factors

LIO-SAM uses **iSAM2**, an incremental optimizer from GTSAM, to update the graph efficiently in real time.

#### Loop Closure Detection

Loop closure is used when the robot returns to a previously visited area. This creates a constraint between non-consecutive keyframes and helps correct accumulated drift.

The default implementation uses radius-based loop detection on keyframes. It avoids selecting nearby timestamps as loop candidates because those frames are already temporally close.

Other loop closure methods, such as Scan Context, can also be integrated to improve place recognition.

## 4. Transform Fusion Node

The Transform Fusion node produces a smooth and high-frequency odometry output by combining:

- High-rate IMU preintegrated odometry, around 200 Hz
- Lower-rate map optimization odometry, around 10 Hz

When a new optimized LiDAR pose is received, the node uses it as a correction. Then, for each subsequent IMU message, it propagates the corrected pose forward using IMU increments.

This produces a high-frequency odometry stream that combines:

- The smooth motion prediction of the IMU
- The long-term accuracy of LiDAR mapping

This output is especially useful for real-time navigation and control.

## Loop Closure in LIO-SAM

Loop closure is an important part of LIO-SAM because LiDAR-inertial odometry still accumulates drift over time. When the robot revisits a place, LIO-SAM can detect the overlap and add a loop closure constraint to the factor graph.

After optimization, the previous trajectory and map are adjusted to reduce accumulated error. This helps produce a more globally consistent map.

## Advantages of LIO-SAM

LIO-SAM has several advantages for robotics and autonomous systems.

### 1. Multi-Sensor Fusion Ready

Because LIO-SAM is based on a factor graph, additional sensors can be integrated by adding new factors with appropriate noise models.

Possible additional measurements include:

- GPS
- Wheel odometry
- Landmark observations
- Visual odometry
- Map-based observations

This makes LIO-SAM flexible for research and real-world robotic platforms.

### 2. Modular Node Design

The system is divided into clear modules:

- Image Projection
- IMU Preintegration
- Map Optimization
- Transform Fusion

This modular structure makes the code easier to understand, debug, modify, and extend.

### 3. Real-Time iSAM2 Optimization

LIO-SAM uses GTSAM's iSAM2 incremental optimizer. Instead of optimizing the full graph from scratch every time, iSAM2 updates only the affected parts of the graph.

This makes LIO-SAM suitable for real-time applications.

### 4. Flexible Loop Closure Integration

Although LIO-SAM includes a default radius-based loop closure method, other approaches can also be added. For example, Scan Context can be used to improve loop detection in large-scale environments.

### 5. Support for Optional Measurements

LIO-SAM can include extra measurements depending on sensor availability. If GPS is available, it can be added as a GPS factor. If GPS is unavailable, the system can still run with LiDAR and IMU.

## Limitations and Possible Improvements

Although LIO-SAM is powerful, there are still areas where it can be improved.

### 1. Dataset Diversity

LIO-SAM could be evaluated on a wider range of datasets to better analyze its robustness, computational performance, and scalability.

### 2. Computational Complexity

For very large maps, graph optimization and loop closure can become computationally expensive. If this becomes a limitation, fixed-lag smoothing could be used to keep only a recent window of states active in the optimization.

### 3. Improved Loop Closure

The default loop closure method is simple and effective, but more advanced place recognition methods such as Scan Context may improve performance in large and repetitive environments.

### 4. Sensor Compatibility

The original LIO-SAM is mainly designed for a 9-axis IMU. Supporting 6-axis IMU sensors requires additional modifications or specific variants of the system.

## Summary

LIO-SAM is a tightly coupled LiDAR-inertial odometry framework that combines LiDAR, IMU, GPS, and loop closure information using factor graph optimization.

Its main strengths are:

- Accurate LiDAR-inertial odometry
- High-frequency state estimation
- Modular ROS node design
- Real-time iSAM2 optimization
- Support for global optimization and loop closure
- Flexibility for adding new sensor factors

Because of its modular design and factor graph foundation, LIO-SAM is not only a strong SLAM framework but also a useful research platform for developing custom multi-sensor fusion systems.
