## Introduction

Sensor fusion is a core component in modern robotics, autonomous driving, and SLAM systems. Robots usually rely on multiple sensors such as LiDAR, IMU, GPS, wheel odometry, and cameras to estimate their position and motion accurately.

Traditional sensor fusion methods, such as the Extended Kalman Filter (EKF), use a recursive prediction-correction approach. While EKF-based systems are useful and widely used, they also have some limitations. They linearize nonlinear models only once per step, which can lead to accumulated errors over time. They also marginalize past states, making it difficult to correct previous estimates after detecting a loop closure. In addition, adding new sensor types often requires redesigning parts of the filter architecture.

A factor graph formulation provides a more flexible and powerful alternative. Using the Georgia Tech Smoothing and Mapping library, commonly known as **GTSAM**, sensor fusion can be formulated as a nonlinear least-squares optimization problem over the robot trajectory.

Instead of estimating only the current state, factor graphs optimize multiple states together, allowing the system to improve both current and previous estimates when new information becomes available.

---

## What Is a Factor Graph?

A factor graph is a probabilistic graphical model that represents the relationship between unknown variables and sensor measurements.

In robotics, the unknown variables are usually robot states such as:

- Pose
- Velocity
- IMU bias
- Landmark position
- Sensor calibration parameters

The measurements are represented as factors. Each factor adds a constraint between one or more variables.

For example:

- Wheel odometry creates a constraint between two consecutive poses.
- GPS creates an absolute position constraint.
- IMU preintegration creates a constraint between pose, velocity, and bias states.
- Loop closure creates a constraint between two non-consecutive poses.

In simple terms:

> Variables represent what we want to estimate, while factors represent what the sensors tell us.

---

## Why Factor Graphs Are Useful for Sensor Fusion

Factor graphs provide several important advantages for sensor fusion.

### 1. Modular Sensor Integration

One of the strongest advantages of factor graphs is modularity. Each sensor can be added as a separate factor without changing the full system architecture.

For example:

- IMU data can be added using an IMU factor.
- GPS data can be added using a GPS factor.
- Wheel odometry can be added using a between factor.
- Map-matching corrections can be added using a prior factor.
- Loop closure can be added using another between factor.

This makes the system easier to extend compared to traditional filters.

---

### 2. Global Optimization

Filtering methods such as EKF mainly estimate the current state recursively. In contrast, factor graphs solve a smoothing problem.

This means the optimizer can adjust several states in the trajectory, not just the latest one. When a loop closure or GPS correction is added, the graph can re-optimize the trajectory and reduce accumulated drift.

This is especially useful in SLAM systems, where drift correction is very important for building a consistent map.

---

### 3. Better Handling of Loop Closure

Loop closure happens when a robot returns to a previously visited location. In EKF-based systems, correcting old poses after loop closure is difficult because previous states are usually marginalized.

In a factor graph, loop closure is naturally represented as a constraint between two non-consecutive poses. When this constraint is added, the optimizer updates the related poses and improves the global trajectory.

---

## Why Use GTSAM?

Several optimization libraries can be used for robotics and SLAM, such as Ceres Solver, g2o, SymForce, and GTSAM. However, GTSAM has several advantages that make it especially suitable for factor graph-based sensor fusion.

### 1. Incremental Optimization with iSAM2

A standard nonlinear optimizer often solves the full graph again whenever a new measurement is added. This can become expensive for real-time systems.

GTSAM provides **iSAM2**, which stands for Incremental Smoothing and Mapping. iSAM2 updates the solution incrementally instead of solving the complete problem from scratch every time.

It uses a Bayes tree structure and only recalculates the parts of the graph affected by new measurements.

This makes GTSAM suitable for online robotics applications where new sensor data arrives continuously.

---

### 2. Fixed-Lag Smoothing

For long-duration missions, keeping the full trajectory in memory can become computationally expensive.

GTSAM provides fixed-lag smoothing, where only a recent sliding window of states is optimized. Older states are marginalized out of the active optimization window.

This provides a balance between:

- The accuracy of smoothing
- The bounded memory usage of filtering

Fixed-lag smoothing is useful for real-time autonomous driving, mobile robotics, and long-term navigation.

---

### 3. Strong Robotics Ecosystem

GTSAM is widely used in robotics research and real-world SLAM systems. It is used in systems such as:

- LIO-SAM
- Visual-inertial odometry systems
- LiDAR-inertial SLAM systems
- Multi-sensor fusion frameworks

Because of this, GTSAM is a strong foundation for developing custom sensor fusion pipelines.

## Core GTSAM Factors for Sensor Fusion

GTSAM provides many predefined factors that can be used to build complex sensor fusion systems. This reduces the need to manually derive Jacobians for every sensor model.

---

## 1. Prior Factor

A prior factor constrains a single variable to a known value. It is commonly used to initialize the graph and define the coordinate frame.

For example, the first robot pose can be fixed at the origin.

```cpp
gtsam::PriorFactor<gtsam::Pose3>(
    first_pose_key,
    gtsam::Pose3::identity(),
    fixed_noise_model
);
```

This prevents the graph from floating freely in space and provides a reference frame for the optimization.

A prior factor can also be used to add absolute corrections from map matching, localization, or external measurements.

```cpp
gtsam::PriorFactor<gtsam::Pose3>(
    current_pose_key,
    map_matched_pose,
    measurement_noise_model
);
```

---

## 2. Between Factor

The between factor is one of the most commonly used factors in GTSAM. It constrains the relative transformation between two states.

It can be used for:

- Wheel odometry
- LiDAR odometry
- Visual odometry
- Constant velocity motion models

```cpp
gtsam::BetweenFactor<gtsam::Pose3>(
    previous_pose_key,
    current_pose_key,
    relative_transform,
    odometry_noise_model
);
```

This factor tells the optimizer how the robot moved from one pose to the next.

---

## 3. Loop Closure Factor

A loop closure factor is usually implemented as a special case of the between factor. The difference is that it connects two non-consecutive poses.

```cpp
gtsam::BetweenFactor<gtsam::Pose3>(
    old_pose_key,
    current_pose_key,
    loop_closure_transform,
    loop_closure_noise_model
);
```

This factor tells the optimizer that the robot has returned to a previously visited location. After adding this constraint, the graph can correct accumulated drift and improve global consistency.

---

## 4. GPS Factor

A GPS factor provides an absolute position constraint. It is useful for reducing global drift, especially in outdoor environments.

```cpp
gtsam::GPSFactor(
    pose_key,
    gtsam::Point3(gps_x, gps_y, gps_z),
    gps_noise_model
);
```

The GPS factor constrains the position of the robot but does not directly constrain its orientation.

This makes it useful when GPS provides global position information, while orientation is estimated from IMU, LiDAR, or visual odometry.

---

## 5. IMU Preintegration Factor

IMUs provide high-frequency measurements of angular velocity and linear acceleration. A simple approach would be to integrate every IMU measurement directly, but this is computationally expensive and can overwhelm the optimizer.

IMU preintegration solves this problem by compressing many IMU measurements between two keyframes into a single relative motion constraint.

This allows the optimizer to use high-frequency IMU information without adding every individual IMU measurement as a separate optimization variable.

Example IMU integration:

```cpp
imuIntegrator->integrateMeasurement(
    gtsam::Vector3(
        thisImu.linear_acceleration.x,
        thisImu.linear_acceleration.y,
        thisImu.linear_acceleration.z
    ),
    gtsam::Vector3(
        thisImu.angular_velocity.x,
        thisImu.angular_velocity.y,
        thisImu.angular_velocity.z
    ),
    dt
);
```

Prediction using the preintegrated IMU measurement:

```cpp
gtsam::NavState currentState =
    imuIntegrator->predict(prevStateOdom, prevBiasOdom);
```

The IMU factor is especially important in LiDAR-inertial and visual-inertial systems because it provides smooth high-rate motion estimation between lower-frequency sensor updates.

## Example Sensor Fusion Structure

A simple factor graph-based fusion system may include the following factors:

```text
X0 ---- X1 ---- X2 ---- X3 ---- X4
 |      |       |       |       |
Prior  Odom    Odom    Odom    GPS
        \\               /
         Loop Closure
```

Where:

- `X0, X1, X2, ...` are robot poses.
- Odometry factors connect consecutive poses.
- GPS provides absolute position correction.
- Loop closure connects non-consecutive poses.
- Prior factor anchors the graph.

This structure allows the robot trajectory to be optimized globally.

## Advantages of GTSAM for Sensor Fusion

GTSAM provides several practical advantages:

- Flexible sensor integration
- Support for nonlinear optimization
- iSAM2 for real-time incremental updates
- Fixed-lag smoothing for bounded computation
- Strong support for SLAM and robotics applications
- Predefined factors for common sensor models
- Easy integration of loop closure and GPS corrections

These features make GTSAM suitable for building robust multi-sensor fusion systems.

---

## Limitations of GTSAM

Although GTSAM is powerful, it also has some drawbacks.

### 1. Computational and Memory Cost

Full batch optimization can become expensive as the number of states and factors increases. For long trajectories, memory and computation can become a problem.

This can be reduced using:

- iSAM2
- Fixed-lag smoothing
- Keyframe selection
- Marginalization
- Efficient factor design

---

### 2. Learning Curve

GTSAM is not as simple as some general nonlinear least-squares libraries. Understanding variables, factors, noise models, keys, and graph optimization requires time.

For beginners, the API can feel complex at first.

---

### 3. Version and Build Issues

GTSAM depends heavily on libraries such as Eigen and Boost. Sometimes, different versions of these dependencies can create build or runtime issues.

In addition, changes between GTSAM versions may break older codebases. This is important when adapting existing systems such as LIO-SAM or other SLAM frameworks.

---

## Summary

Factor graph-based sensor fusion provides a powerful alternative to traditional filtering methods such as EKF. Instead of only estimating the current state, factor graphs optimize multiple states together and allow past estimates to be corrected when new information becomes available.

GTSAM is one of the most widely used libraries for factor graph optimization in robotics. It provides important tools such as iSAM2, fixed-lag smoothing, IMU preintegration, GPS factors, between factors, and prior factors.

For robotics applications such as SLAM, LiDAR-inertial odometry, visual-inertial navigation, and autonomous driving, GTSAM offers a flexible and modular framework for building robust sensor fusion systems.

However, it also comes with challenges such as computational cost, a steeper learning curve, and dependency/version management issues.

Overall, GTSAM is a strong choice for developing advanced sensor fusion systems where accuracy, modularity, and global consistency are important.
